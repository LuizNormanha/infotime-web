## 3. Tier 1 — Implementações rápidas (dados já disponíveis)

### 3.1 Aging de recebíveis

**O que é:** Visão da carteira de contas a receber segmentada por faixas de atraso: corrente (não vencido), 1–30 dias, 31–60 dias, 61–90 dias e 90+ dias. Métrica central de gestão de crédito usada por CFOs.

**Por que é diferencial:** Nenhum concorrente brasileiro de laboratório tem. Internacionalmente é feature central de AR moderno: HighRadius, Gaviti, Tesorio. Empresas que monitoram aging reduzem o DSO em até 40% (Billtrust, 2025).

**Dados necessários:** `infotime_lancamento_receita` — campos `data_previsao`, `valor_previsao`, `id_situacao_documento`, `id_cliente`, `nome_agente`. **Sem alteração de banco.**

**Endpoint:**
```
GET /api/financeiro/aging
Query: { periodo?: 'mes' | 'trimestre' | 'ano', id_empresa?: number }

Resposta:
{
  corrente:          { total: number, qtd: number, itens: AgingItem[] }
  ate_30:            { total: number, qtd: number, itens: AgingItem[] }
  de_31_a_60:        { total: number, qtd: number, itens: AgingItem[] }
  de_61_a_90:        { total: number, qtd: number, itens: AgingItem[] }
  acima_90:          { total: number, qtd: number, itens: AgingItem[] }
  total_carteira:    number
  percentual_atraso: number
  evolucao_mensal:   AgingEvolucao[]
}
```

**SQL base:**
```sql
SELECT
  lr.id_lancamento_receita,
  lr.nome_agente,
  lr.valor_previsao,
  lr.data_previsao,
  (CURRENT_DATE - lr.data_previsao::date)  AS dias_atraso,
  CASE
    WHEN lr.data_previsao::date >= CURRENT_DATE                              THEN 'corrente'
    WHEN (CURRENT_DATE - lr.data_previsao::date) BETWEEN 1  AND 30          THEN 'ate_30'
    WHEN (CURRENT_DATE - lr.data_previsao::date) BETWEEN 31 AND 60          THEN 'de_31_a_60'
    WHEN (CURRENT_DATE - lr.data_previsao::date) BETWEEN 61 AND 90          THEN 'de_61_a_90'
    ELSE 'acima_90'
  END AS faixa
FROM infotime_lancamento_receita lr
WHERE lr.id_tenacidade         = :tenant_id
  AND lr.id_situacao_documento NOT IN (4, 24)   -- excluir Pago e Pago Parcial
  AND lr.data_previsao         IS NOT NULL
ORDER BY dias_atraso DESC;
```

**Frontend:**
- `AgingChart.tsx` — barras horizontais (Recharts), 5 faixas, gradiente verde → vermelho.
- Cada barra clicável abre `ContasReceberList` com `?faixa=ate_30` pré-aplicado.
- Card de tendência: percentual da carteira em atraso vs. mês anterior.

**Posição:** segunda aba do cockpit de gestão integrada.

### 3.2 DRE simplificada no cockpit

**O que é:** Demonstrativo de Resultado do Exercício integrado ao cockpit, atualizado em tempo real. Linha de receita bruta → deduções → receita líquida → custos variáveis → margem de contribuição → custos fixos → EBITDA estimado → resultado líquido.

**Por que é diferencial:** WorkLab tem DRE separada e estática. Integrar ao cockpit com dados ao vivo e drill-down por categoria é diferencial imediato para o segmento de laboratórios.

**Alterações em tabela legada `infotime_plano_conta` — novas colunas snake_case:**

**Job BullMQ — `dunning.processor.ts`:**
```typescript
// Cron: todo dia às 08:00 (configurável por tenant)
// Para cada tenant ativo:
//   1. Buscar lançamentos em atraso (data_previsao < hoje, situação ≠ pago)
//   2. Para cada lançamento, calcular dias_atraso = CURRENT_DATE - data_previsao
//   3. Identificar regra aplicável (por prioridade: id_cliente > id_tipo_agente > global)
//   4. Encontrar etapa da régua cujo dias_apos_vencimento == dias_atraso
//   5. Verificar se já foi enviada (infotime_regua_cobranca_envio para esse lançamento + etapa)
//   6. Se não: enfileirar no worker do canal (email/whatsapp/sms)
```

**Algoritmo de matching:**
```
Para cada linha PENDENTE do extrato:
  1. Filtrar lançamentos com valor igual (±R$ 0,01)
  2. Filtrar por data ±3 dias úteis
  3. Se CNPJ disponível: priorizar lançamentos do mesmo CNPJ
  4. Calcular confiança:
     valor exato + data exata + CNPJ   → 95%
     valor exato + data ±1 dia         → 85%
     valor exato + data ±3 dias        → 70%
     valor ±1%   + data ±3 dias        → 50%
  5. confiança >= 85%: conciliar automaticamente (status='CONCILIADO_AUTO')
  6. confiança < 85%:  listar como sugestão para conciliação manual
```

**Modelo de score:**
```
score_base = 100

PENALIZAÇÕES (cumulativas):
  dia_medio_atraso × 0.5              → ex: média 10 dias → −5 pts
  percentual_em_atraso × 0.3          → ex: 30% atrasam → −9 pts
  maior_atraso_6m × 0.1               → cap: −20 pts
  inadimplência definitiva: −10 cada  → cap: −30 pts

BÔNUS:
  pagamentos antecipados: +2 por ocorrência → cap: +10 pts
  nenhum atraso nos últimos 90 dias: +10 pts
  relacionamento > 12 meses sem inadimpl.: +5 pts

score_final = GREATEST(0, LEAST(100, score_base))
classificacao:
  80–100 → VERDE   (bom pagador)
  50–79  → AMARELO (atenção)
  0–49   → VERMELHO (alto risco)
```

**Job BullMQ:** executa todo domingo às 02:00 — recalcula todos os clientes com lançamentos nos últimos 12 meses e desnormaliza em `infotime_cliente`.

**Regras de detecção:**
```
Regra 1 — Valor anômalo por fornecedor:
  valor > média_fornecedor_12m + (2 × desvio_padrão)
  Severidade: MEDIA se 2-3σ, ALTA se > 3σ

Regra 2 — Pagamento duplicado suspeito:
  mesmo id_fornecedor + valor igual (±R$0,01) + data ±3 dias + não é parcela
  Severidade: ALTA

Regra 3 — Fornecedor novo com valor alto:
  primeiro lançamento do fornecedor + valor > threshold (padrão: R$5.000)
  Severidade: MEDIA

Regra 4 — Valor recebido diverge do previsto:
  |valor_realizacao - valor_previsao| / valor_previsao > 10%
  Severidade: BAIXA (10-20%), MEDIA (>20%)

Regra 5 — Lançamento em horário suspeito:
  data_inclusao entre 22:00 e 06:00 + valor > ticket_médio_usuário × 3
  Severidade: MEDIA
```

## 9. Padrões obrigatórios em toda implementação

- Ler **todos** os arquivos em `/mnt/project/` antes de qualquer módulo.
- `id_tenacidade` sempre do JWT — nunca do body, path ou query params.
- Toda tabela nova: prefixo `infotime_` + snake_case + campos de auditoria + `GENERATED BY DEFAULT AS IDENTITY`.
- Campos booleanos: `bpchar(1)` com `'S'`/`'N'`, nunca `BOOLEAN`.
- Toda listagem: paginação server-side com `ListQuery` / `ListResponse<T>`.
- Todo formulário: `FormShell` + `TabView` + `FormFooter` + `ActionsBar`.
- Todo endpoint: Zod na rota → lógica no service → query no repository.
- Jobs BullMQ: registrar início, fim, total processado, erros e duração em log JSON estruturado.
- Webhooks externos (PIX, WhatsApp): validar assinatura HMAC **antes** de qualquer processamento.
- Credenciais sensíveis (PIX `client_secret`): armazenar criptografadas com AES-256.
- Multi-tenant: todo `SELECT`, `INSERT`, `UPDATE` e `DELETE` filtra por `id_tenacidade` do JWT.
- Uploads: nunca salvar no servidor local — storage externo conforme `PADRAO_UPLOAD_ARQUIVOS.md`.
